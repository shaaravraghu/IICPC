================================================================================
IICPC DISTRIBUTED BENCHMARKING & HOSTING PLATFORM
Target: Signal Pipeline Evaluation
Stack: Kubernetes | Kafka | Rust | TimescaleDB
================================================================================

1. SUBMISSION & SANDBOXING ENGINE (The Gatekeeper)
--------------------------------------------------------------------------------
Contestants upload their signal pipeline source/binaries. The platform must containerize and deploy these submissions in strictly isolated environments to prevent malicious code execution and ensure fair resource allocation (e.g., CPU pinning, strict memory limits). 

- Implementation: A Rust-based API receives the upload and dynamically generates a Kubernetes Job. 
- Isolation: The contestant's pipeline runs in an unprivileged, network-restricted K8s Pod (Sandbox).
- Interface: The Sandbox communicates exclusively via predefined Kafka topics, preventing external network access while allowing the platform to ingest the pipeline's trading signals.

2. EVENT STREAMING LAYER (The Nervous System)
--------------------------------------------------------------------------------
Apache Kafka acts as the backbone for inter-service communication and deterministic replay. The event-driven handoff between stages ensures the system remains decoupled and highly concurrent.

- Core Topics:
  * `sys.market_data.live`: Ingestion of historical/mock live market data.
  * `stage1.technical.out`: Tickers surviving the Technical filter.
  * `stage2.fundamental.out`: Tickers surviving the Fundamental filter.
  * `stage3.sentiment.out`: Ranked assets with neural network weights.
  * `eval.contestant.signals`: Trade candidates generated by the contestant Sandbox.
  * `sys.telemetry.metrics`: Raw latency, throughput, and correctness logs.

3. DISTRIBUTED BOT FLEET (The Swarm)
--------------------------------------------------------------------------------
You must build a scalable traffic generation service capable of spawning thousands of distributed bots. Instead of monolithic services, these are deployed as decentralized Rust agents acting as a swarm, subscribing to Kafka topics, and processing metrics with high algorithmic efficiency.

- Stage 1: Technical Analysis Agents
  * Subscribe to `sys.market_data.live`.
  * Execute multiple trigger initiators (20 metric groups like ADX, MACD, etc.).
  * Filter out the largest chunk of the asset universe to focus on a niche segment.
  
- Stage 2: Fundamental Analysis Agents
  * Subscribe to `stage1.technical.out`.
  * Validate potential, consistency, and reliability (e.g., ROIC, FCF Margin) rather than following market trends.
  * Forward surviving assets to Stage 3.

- Stage 3: Sentiment Analysis Agents
  * High-compute final validation layer.
  * Each asset is scored between -5 and +5 across 15 dimensions. 
  * The final ranking utilizes a weighted sum model where the total neural network weight across all dimensions is 100%. The composite sentiment score $S$ for an asset is calculated as $S = \sum_{i=1}^{15} w_i \cdot s_i$, where $w_i$ is the parameter weight and $s_i$ is the dimension score.

- Stage 4: Strategy Execution Simulator (Load Generator)
  * Acts as the core interaction engine, taking the finalized assets and bombarding the contestant's system with concurrent orders to simulate peak market volatility. 
  * Simulates trades across Intra-day, Short-term, Medium-term, and Long-term horizons.

4. TELEMETRY, VALIDATION & LEADERBOARD (The Judge)
--------------------------------------------------------------------------------
A low-latency tracking system monitoring the interactions between the bots and the contestant's exchange. 

- Ingestion: A dedicated Rust service consumes `sys.telemetry.metrics` from Kafka.
- Storage: TimescaleDB stores the time-series data, enabling complex window functions and fast aggregations for scoring.
- Metrics Tracked: It must accurately measure order acknowledgment time (p50, p90, and p99 latencies), maximum transactions per second (TPS) handled before failure, and validation of price-time priority and fill accuracy.
- Deterministic Replay: Because all inputs and K8s scheduler states are logged in Kafka, the judge can replay any execution identically to verify algorithmic correctness.
